# PAYMENT-DOUBLE-FAULT-RECOVERY — 상황 케이스 카탈로그

> 목적: 이전 discuss/plan 산출물(F/D/C 라벨, 대안 B/C, Round 1~4 판정)은 전부 폐기한다.
> 이 파일은 "시스템이 복구해야 할 상황"만 자연어로 나열한 재료다. 라벨·일련번호 없이 케이스 자체로 식별한다.
> 재시작된 discuss는 이 카탈로그를 입력으로 삼아, 각 케이스마다 **"현재 코드에서 복구 가능한가? / 어떤 방어선이 필요한가?"**를 케이스별로 답한다.

---

## 사용 규칙

- 케이스는 **상황 묘사 + 시스템이 해야 할 일**만 쓴다. 해결 방법은 여기 쓰지 않는다.
- 일련번호 금지. 케이스를 가리킬 때는 헤더 문구를 그대로 인용한다.
- 이전 discuss의 결정(대안 C, D9 보수적 디폴트, ALREADY_PROCESSED 분기 등)은 참고 금지.
- 기존 코드 사실(`RETRYING` 상태 존재, `getStatusByOrderId` 포트 존재, `PaymentStatus.UNKNOWN` 존재, Toss idempotency-key는 orderId pass-through)만 사실 재료로 사용.

---

## 케이스

### 사용자 요청이 PG에 도달하기 전에 서버가 죽었다
- 로컬 결제 이벤트는 처리 중 상태로 남았다.
- PG에는 이 주문에 대한 결제가 **아직 존재하지 않는다**.
- 시스템이 해야 할 일: 같은 주문에 대해 PG에 승인을 다시 시도한다. 일정 한도 내에서 재시도 가능해야 한다.

### PG까지 요청은 갔지만 응답을 받기 전에 서버가 죽었다
- PG 쪽 결과는 승인/실패/진행중 어느 것이든 가능하다.
- 로컬은 처리 중 상태.
- 시스템이 해야 할 일: PG 쪽 실제 상태를 먼저 확인한 뒤 그 결과에 따라 행동을 결정한다. 무조건 재승인 요청하면 안 된다.

### PG는 승인 성공했는데 로컬 DB 반영 전에 서버가 죽었다
- PG 상태: 승인 완료 + 승인 시각 존재.
- 로컬: 처리 중 상태.
- 시스템이 해야 할 일: 로컬을 승인 완료로 전환한다. PG에 확인/승인 요청 추가 호출 금지.

### PG는 승인 성공 상태인데 로컬도 이미 승인 완료로 박혀 있다
- 과거 반영이 됐는데 이후 뭔가가 outbox/후속 처리만 재실행시킨 상황.
- 시스템이 해야 할 일: 결제 이벤트 상태 머신을 다시 건드리지 않는다. 후속 처리만 멱등하게 마무리.

### PG는 승인 성공인데 승인 시각이 비어있다
- 로컬에 "승인 완료 + 승인 시각 null"이 박힐 가능성이 있다.
- 시스템이 해야 할 일: 이 상태 자체를 만들지 않는다. 만들어지려는 순간 차단 — 승인 시각 없으면 완료 전환 금지.

### PG가 취소·중단·만료·부분취소 상태를 보고한다
- 로컬은 아직 처리 중 또는 이미 실패 상태일 수 있다.
- 시스템이 해야 할 일: 실패 보상 경로로 간다. 재고 복구는 정확히 한 번만 — 이미 실패 처리된 건에 대해 다시 복구하면 안 된다.

### PG에 주문을 조회했는데 "없는 주문"이라고 응답한다
- 로컬 처리 중 상태로 남아있는데 PG에는 흔적이 없다.
- 시스템이 해야 할 일: "첫 번째 케이스"(요청이 도달 전 사망)와 동일하게 취급한다. 승인 시도. 단, 진짜 "없음"과 "일시 오류로 없음처럼 보임"을 구분해야 한다.

### PG 조회가 타임아웃/5xx/네트워크 오류로 실패한다
- 지금 이 순간 PG 상태를 알 수 없다.
- 시스템이 해야 할 일: 판단 불가로 간주하고 다음 주기에 다시 시도한다. 재시도 한도 안에서 몇 번이든. 로컬 상태 전이 금지.

### PG가 우리가 모르는 상태 문자열을 반환한다
- enum 매핑이 안 된다.
- 시스템이 해야 할 일: 조용히 흡수하지 않는다. 판단 불가로 돌리거나 명시적으로 실패시켜 사람이 보게 한다. "알 수 없음"을 정상 흐름의 한 갈래로 두지 않는다.

### 같은 처리 중 건을 두 워커가 동시에 잡으려 한다
- 다중 인스턴스 환경에서 스케줄러가 겹친다.
- 시스템이 해야 할 일: 정확히 하나만 이긴다. 진 워커는 해당 건을 즉시 포기.

### 승인 성공했는데 같은 건에 대해 나중에 또 승인 루틴이 돌았다
- 재시도 로직/복구 로직/수동 개입 등 이유는 여러 가지.
- 시스템이 해야 할 일: 상태 머신 재진입을 거부한다. 이미 완료된 건은 완료만 재확인하고 끝.

### 보상 트랜잭션이 중간에 서버 장애로 부분 실행됐다
- 예: 재고는 복구됐는데 결제 이벤트 상태 전이 전에 죽었다. 또는 반대.
- 시스템이 해야 할 일: 재기동 후 보상을 안전하게 재개한다. 재고 이중 복구 금지, 결제 상태는 멱등 전이.

### 재시도 한도 내에서 몇 번을 해도 PG가 계속 판단 불가 상태를 준다
- 타임아웃/5xx가 지속.
- 시스템이 해야 할 일: 한도 소진 시 격리 상태로 이동. 자동 실패 확정 금지 — 사람이 보도록.

### 재시도 한도 내에서 순수 1차 장애(없는 주문) 승인 재시도가 모두 실패했다
- 승인 요청을 N번 보냈는데 다 일시 에러 등으로 돌아왔다.
- 시스템이 해야 할 일: 한도 소진 시 결제 실패 확정 + 재고 복구. 사용자에게는 실패로 통지.

## 기존 산출물에서 건질 사실 (참조용, 라벨 없이)

- `PaymentEventStatus.RETRYING` 및 `PaymentEvent.toRetrying()`은 이미 존재. 재시도 시 상태 머신의 중간 상태로 쓸 수 있다.
- `PaymentGatewayPort.getStatusByOrderId` 포트와 `TossPaymentGatewayStrategy.getStatusByOrderId` 구현체는 이미 존재. 사용처가 없을 뿐.
- `PaymentStatus.UNKNOWN` enum 값이 현재 존재한다. 매핑 불가 시 fallback으로 사용 중.
- Toss의 `Idempotency-Key` 헤더는 `orderId`를 그대로 pass-through 한다. 동일 key로 재호출 시 원 응답을 반환.
- `OutboxProcessingService.process`는 현재 `claimToInFlight` → `confirmPaymentWithGateway` 호출 구조. `getStatusByOrderId` 경로 미사용.
- 현재 복구 사이클은 매 틱마다 `confirm`을 재호출하는 방식이다. getStatus 선행 조회는 없다.
- 기본 복구 테스트 커버리지(재시도 소진 → 보상, NON_RETRYABLE 즉시 보상, 보상 성공, PaymentEvent 로드 실패 → incrementRetryOrFail, IN_FLIGHT 타임아웃 복구)는 존재한다.
