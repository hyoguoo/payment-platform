# GitHub 컨벤션 (이슈 · 브랜치 · PR)

이슈/브랜치/PR을 만들거나 갱신하는 모든 작업의 표준. 메인 스레드가 직접 수행한다.

**도구**: `mcp__github__*` 툴 우선 (`issue_write`, `create_pull_request`, `update_pull_request`). gh CLI는 인증이 확인된 경우에만.

**어체**: 이슈/PR의 모든 한국어 문장은 명사형 종결 또는 `~다` 어체. `~습니다` 류 경어체 금지.

## Step 1 — 이슈 생성 (discuss 완료 시)

- 제목: **한글**로 무슨 작업인지 읽히게. 토픽 코드(영문 UPPER-KEBAB-CASE)는 브랜치명·파일명 전용 — 제목에 넣지 않는다.
- 본문: **불릿 중심으로 간결하게**. 헤더는 **명사형**(`배경` / `목표` / `범위` / `비범위` / `검증 방법` 등) — 구어체 소제목 금지. "다음 단계" 섹션 금지.
- 내부 식별자(태스크 ID, 결정 ID, 토픽 코드)는 노출하지 말고 내용을 풀어 쓴다. 설계 문서 경로 참조는 1줄로. 필요하면 Mermaid 가능.
- **Labels + Assignees 지정**: 변경 유형 라벨(`enhancement` / `refactor` / `documentation` / `test` / `instruction` 등, 복수 가능) + 작업자 assignee.

## Step 2 — 브랜치

- 브랜치명: `#<issue-number>` (예: `#47`), `main`에서 분기
- 강제 push(`--force`) 금지, 필요 시 `--force-with-lease`

## Step 3 — PR 생성 (ship 단계)

- 조건: ship 체크리스트 통과 AND 로컬 전체 테스트 pass
- 제목: `<type>: <한글 요약>` (70자 이내). 토픽 코드 금지.
- Labels / Assignees: 이슈와 동일 기준
- 본문 템플릿 (한글 명사형 헤더, 불릿 중심):

  ```
  ## 관련 이슈
  Closes #<issue-number>

  ## 개요
  <문제·필요성·해소 방향을 간결히>

  ## 구현 내용
  ### <영역 1 — 예: 도메인>
  - 변경 내용 상세

  ## 상태 머신 / 플로우 (해당 시)
  <Mermaid 다이어그램>

  ## 테스트
  - <어떤 테스트가 어떤 케이스를 커버하는지>
  ```

- **금지**: 파일 경로 추측 (실제 `git diff --stat` 기반으로만 작성) / 영문 헤더(Summary, Changes 등) / 내부 식별자 노출 / 푸터(Generated with 등)
- 시각 보조: 동시성·분기·구조 변화처럼 텍스트로 어려운 내용만 Mermaid/표 사용. 충분히 명확하면 넣지 않는다.

## Step 4 — PR 갱신

- 리뷰 피드백 대응은 **새 커밋** + push. 본문 수정은 `update_pull_request` / `gh pr edit`.
- **히스토리 보존**: 본문을 통째로 교체하지 않는다. 기존 구현 내용·설계 배경·발견된 버그 기록을 보존하고, 새 변경은 기존 섹션 아래에 누적한다. 기준: "처음 보는 리뷰어가 본문만 읽고 전체 변경 흐름을 이해할 수 있는가".
- **리포트성 코멘트는 단일 통합 코멘트로** — 여러 댓글로 분리하지 않는다. 풍부함은 한 코멘트 안의 표/섹션으로.

## 금지

- `main` 직접 push
- lease 없는 force-push
- self-merge — merge는 **사용자가 명시적으로 요청할 때만** 실행
- 커밋 메시지/PR 본문에 비밀 값
