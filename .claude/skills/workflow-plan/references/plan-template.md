# docs/<TOPIC>-PLAN.md 작성 형식

```markdown
# <주제> 구현 플랜

> 작성일: YYYY-MM-DD

## 목표

<한 줄 목표 — 무엇을 완료하면 이 플랜이 끝나는지>

## 컨텍스트

- 설계 문서: [docs/topics/<TOPIC>.md](../topics/<TOPIC>.md)
- 주요 변경 파일: <예상되는 핵심 파일들>

---

## 태스크

### Task 1: <이름> [tdd=false]

**구현**
- 무엇을 어디에 만들지 (파일 경로 포함)

**완료 기준**
- 컴파일 오류 없음
- 기존 테스트 통과

---

### Task 2: <이름> [tdd=true]

**테스트 (RED)**
- 어떤 클래스에 어떤 테스트를 작성할지
- 예: `IdempotencyStoreTest` — `get_존재하는_키_반환`, `save_중복_키_덮어쓰지_않음`
- 사용할 패턴: `@ParameterizedTest` / Mockito BDD style / AssertJ

**구현 (GREEN)**
- 테스트를 통과하는 최소 구현
- 어떤 클래스를 어디에 만들지

**리팩터 (REFACTOR)**
- 정리가 필요하면 기술, 불필요하면 생략

**완료 기준**
- 해당 테스트 클래스 전체 통과
- `./gradlew test` 회귀 없음

---

### Task N: ...
```

---

## 태스크 품질 체크

플랜 작성 후 아래를 점검한다.

- **누락된 의존성**: Task 3이 Task 5에서 만드는 클래스를 사용한다면 순서를 바꾼다.
- **너무 큰 태스크**: "서비스 전체 구현"처럼 뭉뚱그린 태스크는 나눈다.
- **tdd=true인데 완료 기준이 모호**: 어떤 테스트 메서드가 통과해야 하는지 명시한다.
- **Fake 누락**: tdd=true 태스크에서 새 포트를 사용한다면 해당 Fake를 앞 태스크로 추가한다.
