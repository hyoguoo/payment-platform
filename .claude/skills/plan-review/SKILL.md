# Plan Review

구현 전에 계획의 결함을 찾는 것이 목적이다. 코딩 시작 후 발견된 문제는 되돌리는 비용이 크다.

---

## 입력 판별

| 상황 | 처리 |
|------|------|
| 파일 경로 지정 (`path/to/PLAN.md`) | 해당 파일을 읽어 검수 |
| "방금 내용", "위 내용" 등 | 직전 대화의 내용을 검수 대상으로 사용 |
| 내용 직접 붙여넣기 | 붙여넣은 내용을 검수 |
| 아무것도 지정하지 않음 | "어떤 내용을 검수할까요?" 질문 |

컨텍스트가 payment-platform 프로젝트라면 `docs/context/ARCHITECTURE.md`와 `docs/context/TESTING.md`도 참조한다.

---

## 검수 항목

결함 발견 시 등급(🔴/🟡/🟢)과 함께 기록한다.

**1. 설계 일치** — 상위 설계/토픽 문서의 결정이 계획에 그대로 반영되었는가? 명시적으로 제외된 항목이 몰래 포함되지 않았는가?

**2. 아키텍처 규칙** — 헥사고날 레이어를 침범하는 태스크가 없는가? (도메인에 Spring 의존, 포트 위치 오류, 컨텍스트 간 직접 레포지토리 공유 등)

**3. 태스크 완전성** — 포트만 있고 구현체가 없거나, 설정 클래스·마이그레이션·빈 등록이 누락되지 않았는가? execute를 끝냈을 때 목표가 완전히 달성되는가?

**4. 의존성 순서** — 후행 태스크가 아직 만들어지지 않은 클래스를 사용하지 않는가? Fake 구현체는 그것을 사용하는 tdd=true 태스크보다 앞에 있는가?
권장: 포트 → 도메인 → 애플리케이션 → 인프라 → 컨트롤러

**5. TDD 계획** — 비즈니스 로직/상태 전환/엣지 케이스가 있는 태스크는 tdd=true인가? 테스트 클래스명과 메서드(또는 시나리오)가 충분히 명시되었는가?

**6. 완료 기준 명확성** — "잘 동작함" 같은 주관적 표현 없이, 어떤 테스트가 통과해야 하는지 또는 무엇이 성립해야 하는지 객관적으로 판단 가능한가?

**7. 태스크 크기** — 각 태스크가 독립 커밋 단위인가? "전체 서비스 구현"처럼 너무 크거나, 의미 없이 쪼개진 태스크는 없는가?

**8. 결제 도메인 엣지 케이스** *(해당 시)* — 멱등성, 동시성 전략, 트랜잭션 경계, Toss 호출 실패 복구, Sync/Outbox/Kafka 호환성이 필요한 태스크에 명시되었는가?

---

## 등급 기준

| 등급 | 기준 | 조치 |
|------|------|------|
| 🔴 CRITICAL | 설계 자체가 잘못되어 구현하면 전면 재작업 필요 | 논의 복귀 권고 |
| 🟡 MAJOR | plan 구조 오류(순서, 누락, 모호한 명세). 코딩 전 반드시 수정 | 수정 후 재점검 |
| 🟢 MINOR | 구현을 막지 않는 개선 제안 | 메모만, 진행 가능 |

CRITICAL·MAJOR는 의견이 아닌 사실 근거여야 한다. "이 방식이 더 나을 것 같다"는 MAJOR가 아니다.

---

## 보고서 형식

```
## Plan Review 결과

### 요약
🔴 CRITICAL N건 · 🟡 MAJOR N건 · 🟢 MINOR N건

### 🔴 CRITICAL
**C1. <제목>**
위치: ... / 문제: ... / 근거: (왜 discuss를 다시 해야 하는 수준인지)

### 🟡 MAJOR
**M1. <제목>**
위치: ... / 문제: ... / 수정 방향: ...

### 🟢 MINOR
**N1. <제목>**
위치: ... / 제안: ...
```

결함이 없으면 "검수 통과 — 발견된 결함 없음"으로 종료한다.
